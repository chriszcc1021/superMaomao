# GPT-5.5 重写 / 重构清单

> 日期：2026-04-25  
> 项目：《喵喵远征》  
> 技术栈：Godot 4.2.2 + GDScript  
> 目标：用 GPT-5.5 做第二版工程整理，而不是盲目推倒重写。

## 结论

可以用 GPT-5.5 重写一遍，但推荐做“阶段式重构 / 第二版重写”：

- 保留已经跑通的玩法、数值、场景、数据文件和自动测试入口。
- 优先拆分过大的脚本，降低后续继续开发的风险。
- 每完成一个阶段都运行 Godot 解析检查或自动化流程，保证游戏始终可运行。
- 不在没有验证的情况下同时重写营地、战斗、远征和繁育。

## 当前项目观察

- 项目已经是完整 Godot 工程，入口场景为 `res://scenes/camp/CampScene.tscn`。
- 已有 Autoload：`Preloader`、`GameState`、`SceneManager`、`EventBus`、`AutoTest`、`TimeManager`。
- 大循环已经存在：营地、出征地图、战斗、商店、奇遇、结算。
- 自动化测试入口已经存在：`autoload/AutoTest.gd`，通过 `AUTOTEST=1` 环境变量激活。
- 现有规则文档要求每次只实现一个功能、数据与逻辑分离、跨场景通信通过信号。

## 主要风险点

- `scenes/camp/CampScene.gd` 约 1180 行，承担 UI、建筑、猫咪拖拽、日结算、初始选猫等太多职责。
- `scenes/battle/BattleScene.gd` 约 695 行，混合了战斗流程、卡牌、基因弹窗、结算和 HUD 刷新。
- `autoload/GameState.gd` 约 583 行，既保存全局状态，也处理繁育坑位、建筑升级、远征状态等业务逻辑。
- 多个脚本超过项目规则里“单个脚本不超过 300 行”的目标。
- 架构文档仍引用较旧设计版本，当前应以 `docs/游戏设计文档-v3.6.md` 为准。

## GPT-5.5 使用原则

- 先读现有文件，再决定改法。
- 每次只改一个系统，不跨阶段混改。
- 优先做小步提交，每个提交都能解释和验证。
- 不为了“更高级”而添加抽象。
- 如果发现设计文档没写清楚，先记录问题，再向人确认。

## 推荐第一步

下一步不要直接重写全项目。推荐先做：

1. 跑一次当前 Godot / AutoTest 基线。
2. 从 `CampScene.gd` 拆出“初始选猫 overlay”或“建筑 sidebar”其中一个小模块。
3. 跑一次验证。
4. 提交一个小而清楚的 commit。

这会让 5.5 的优势用在架构和长期维护上，而不是把已经能跑的游戏重新打散。
